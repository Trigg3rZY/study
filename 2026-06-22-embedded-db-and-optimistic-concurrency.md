# 持久化层的两个工程选择:嵌入式 DB(pglite)与「用 DB 不变量做乐观并发」

> 技术参考。来自 evidata 的一段实战讨论:为什么元数据用 pglite 不用普通 pg、每个对话都持久化对 DB 有什么要求、以及「后台纠错重跑」和「用户同时追问」并发时会不会出问题。
>
> **一句话主轴:存储不是「先上最强的 DB」,而是按规模选「确定性外壳」的形态——小团队用嵌入式零运维、靠 DB 不变量兜并发,需要规模时再「换驱动」毕业到服务端。**

---

## 1. 嵌入式 DB vs 服务端 DB

**pglite** = 把真正的 PostgreSQL 编译成 WASM,**嵌进应用进程里跑**(像 SQLite 那样):没有独立 server、单连接、落一个文件或纯内存。但它是**真 Postgres**——同样的 SQL、类型、schema、迁移、约束。
**服务端 pg** = 独立网络数据库:多连接、真并发、可扩展、可 HA,代价是你要部署运维。

选嵌入式的理由(不是「凑合」,是「按定位对」):
- **零运维**:自托管小团队跑应用**不用额外搭数据库**,少一个要操心、要备份、要监控的组件。嵌入式 = 把「数据库」从「一个服务」降级成「一个文件」。
- **语义不打折**:因为是真 Postgres,将来要扩容是**换驱动**,不是重写(见 §4)。SQLite/pglite 这类嵌入式的价值就在「单机时零摩擦,长大时不堵死」。
- **测试/CI 纯净**:内存版 → 无容器、hermetic、快。

它的边界(也是「对 DB 的要求」的真正含义):
- **单进程 / 单写者**:几个并发用户、写串行,够用;**高并发、多实例水平扩展、HA** 撑不住。
- **存储随业务增长**:要给数据目录留盘 + 定期备份那个文件;历史多了要保留/归档策略。
- 一旦需要「多个应用实例共享同一份数据」,嵌入式就到头了(文件是进程私有的)——这是毕业到服务端的硬触发器。

> **可迁移原则**:嵌入式 DB 是「单机/单实例 + 零运维」区间的最优解。判断要不要毕业,看**三个硬触发器**:真并发写、多实例共享、HA/容灾。**没撞到就别提前上服务端**(过早引入运维复杂度也是一种债)。

## 2. 关键设计:「自己的账本」和「业务数据」是两个库

evidata 里 pglite **只存应用自己的元数据**(对话、回答版本、连接配置、模型配置……)。用户**真正的业务数据**(被提问的那些数据源)是**另一个独立的、真的 Postgres**,只读连过去。

- **元数据**:量小、结构自己定、要零运维 → 嵌入式。
- **业务数据**:量大、不归你管、要真并发 → 连真 pg(或别的源)。

> **可迁移原则**:别把「应用的账本」和「应用要处理的数据」混在一个库里。两者的规模曲线、所有权、运维要求完全不同。分开后,账本可以用最省事的存储,数据源可以按它本来的样子接。很多「该用什么数据库」的纠结,本质是没把这两类分开。

## 3. 乐观并发:用 DB 不变量兜底,而不是到处加锁

问题场景:后台「纠错重跑」正在给某个对话追加一个新回答版本,**同一时刻**用户又在这个对话里追问——两个写并发,会不会写坏?

evidata 的答案是**不会**,机制值得抄:

**(a) 所有「追加版本」都走同一个事务函数。** 追问 / 重跑 / 纠错重跑,全部 funnel 进一个 `saveAnswer`,它在**一个事务**里:读当前最大 version → 把旧 head 降级 → 插入 `version+1` 作为新 head。**单一入口** = 并发语义只需在一处推理。

**(b) 两条 DB 级不变量,把「不可能的状态」变成数据库拒绝的状态:**
- `UNIQUE(investigation_id, version)` —— 不可能有两个相同版本号。
- **部分唯一索引** `UNIQUE(investigation_id) WHERE is_latest` —— 每个对话**最多一个** `isLatest` head。

这两条让「双 head / 重复版本」**在数据库层就插不进去**,而不是靠应用代码小心翼翼维护。

**(c) 不同后端,同一套约束都成立:**
- **pglite(单连接)**:写天然串行——两个操作不可能真并行,一个先提交,另一个读到新的 max 再干净追加。**没冲突。**
- **真 pg(真并发)**:万一真交错,唯一约束让其中一个事务**失败回滚**(fail-closed,绝不产生坏状态)。这时再给「输的那个」加**重试**或**按 investigation 的 advisory lock**,让它干净重排。

> **可迁移原则**:并发正确性优先**靠数据结构(约束/不变量)**而不是**靠控制流(锁/顺序)**。把「绝不允许的状态」编码成 DB 约束 = 不管哪条代码路径、哪个后端、哪种交错,数据库都替你 fail-closed。锁是补充(消除「报错重试」的毛刺),不是第一道防线。这也是「乐观并发」的精髓:**默认不锁,撞了再处理,用唯一约束当裁判。**

**(d) 机制对了,还要想语义。** 上面保证了「不写坏」,但「该不该写」是另一回事:如果用户已经追问、把对话推到更新的版本了,后台纠错再去重答**那个旧问题**、插一条在底下,会突兀。evidata 的决定:**只在「要修正的那个版本仍是对话当前 head」时才自动重跑;否则只把知识核验为 Verified、不重跑**(知识照样惠及后续提问,但不打扰已经走远的对话)。

> **可迁移原则**:并发设计有两层——**完整性**(别写坏,靠约束)和**意图**(这次写还有意义吗,靠业务条件)。前者数据库帮你,后者只能你自己在领域逻辑里判。两层都想到才算想完。

## 4. 可迁移性:让「嵌入式 → 服务端」是换驱动,不是重写

能在小团队用嵌入式、长大了平滑接服务端,靠的是**纪律**:
- **存储走 port(接口),不走具体实现**:应用只依赖 `MetadataStore` 抽象,`DrizzleStore` 底下是 pglite 还是 node-postgres 是细节。
- **迁移用工具生成 + 双后端都能跑**:schema 改动走 drizzle-kit 生成、镜像进迁移列表;只用两边都支持的特性(部分唯一索引、jsonb、带时区时间戳),**不写 pglite-only 的假设**。
- **切换点收敛到一处**:由 env 决定连嵌入式还是服务端 + 一个 `db:migrate` 脚本。最难的是「让 `transaction()` 的类型同时容纳两种驱动」——提前把这个口子留好(evidata 把它单列成一个 issue,正是因为这是唯一的「细心活」)。

> **可迁移原则**:**「平滑毕业」不是天上掉的,是当初留的口子。** 第一天就用嵌入式 + 真 Postgres 语义 + port 抽象,毕业那天就是换驱动 + 加个迁移脚本;第一天图省事写死了具体 DB 的 API,毕业那天就是重写。**便宜的现在,要为不贵的将来留接口。**

---

## 一页速记

- 存储按规模选形态:**嵌入式(零运维/单机)→ 服务端(并发/多实例/HA)**;触发器是「真并发写 / 多实例共享 / HA」,没撞到别提前上。
- **账本归账本,数据源归数据源**:两类数据的规模和所有权不同,分两个库。
- 并发正确性**先靠不变量**(unique version + 单 head 部分唯一索引 + 单一事务入口),**锁是补充**;pglite 串行、真 pg fail-closed。
- 并发要分**完整性**(DB 兜)和**意图**(领域逻辑兜,如「仍是 head 才重跑」)两层。
- 平滑毕业靠**当初留的口子**:port 抽象 + 工具生成的双后端迁移 + 收敛的切换点。
