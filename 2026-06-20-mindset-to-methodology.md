# 从心智到方法论到工程实践:一张落地地图

> 副标题:把"做事"分三层——**心智(为什么)→ 方法论(把心智变成可重复的"如果…就…")→ 工程实践(具体手艺/配置)**。光有心智不会自动变成行动,光有实践不知道何时该用;**方法论是那座桥**。本篇给一张把心智落到实践的地图,并教你自己造桥。

## 0. 一句话定位

> **心智是"为什么重要",工程实践是"具体怎么做",中间缺了方法论就会断层:** 你认同"verification 很重要"(心智),却不知道明天该改什么(实践),因为没人把它翻译成一句**可执行、有触发条件**的规则(方法论)。方法论 = **把 why 编译成 repeatable how。**

---

## 1. 为什么必须有中间这层

两种常见的"断层":

- **只有心智**:读了一堆"判断力很重要""要重视验证",点头称是,然后行为毫无变化——因为心智是**陈述句**,不是**指令**。
- **只有实践**:照着教程配了 CI、装了工具,但不知道**为什么、什么时候该加、什么时候是过度**——于是要么漏配,要么 cargo-cult(抄了形式没抄到判断)。

方法论把两者接上:它是一条 **"如果 X,就 Y"** 的规则,既带着心智的 why,又能直接触发实践的 how。

> 例:心智"verification 缩短判断延迟" → 方法论"**如果**一种错误能被自动检测,**就**把它做成几分钟内出结果的必需门控;产品层**如果**一周没真用过产品,**就**强制跑一条端到端" → 实践"CI 必需 job + CodeQL;每周 try + 漂移 tripwire"。

## 2. 落地地图(核心)

把我们讨论过的心智,逐条翻译成方法论与具体实践。**这张表就是"把心智落实到方法论"本身。**

| 心智(为什么) | 方法论(如果…就…) | 工程实践(具体落地) |
|---|---|---|
| **执行可外包,意图不可** | 凡携带"做什么/什么算好"的决定,先外化成一个 AI 要对它负责的契约 | PRD、spec、AGENTS.md、PR 描述作为契约 → [产品判断](2026-06-20-product-judgment.md) |
| **verification 是延迟的解药** | 能自动检测的错,就做成几分钟的必需门控;产品层别等 milestone 才验 | 必需 CI(typecheck/lint/test/集成/smoke)、CodeQL;产品层每周 try → [交付外壳](2026-06-20-delivery-harness.md) |
| **把判断编译进结构** | 凡是你反复做的同一类判断,就把它变成代码/配置里的护栏 | ruleset、lint 规则、trust boundary("AI 提议、应用校验") → [Harness](2026-06-18-harness-engineering.md) |
| **单向门自己拥有** | 难回退/定义身份的决定,设一道"需要人裁决"的关卡 | `needs-design` 标签、design-before-code、`required_review_thread_resolution` → [决策纪律](2026-06-20-deciding-with-ai.md) |
| **用 AI 当弹药、不当结论** | 在重要决策上,不问"怎么做",改问"反方/选项+trade-off/steelman 我没选的" | 固定的提问模板;PR 里留"为什么不是另一个" → [决策纪律](2026-06-20-deciding-with-ai.md) |
| **分解要 file-disjoint** | 并行任务就切到不同包/目录,标注 Touches | 任务按包切、worktree 一对一、issue 模板的 `Touches` 字段 → [多 agent](2026-06-20-agentic-multi-agent-workflows.md) |
| **漂移靠诱人加法发生** | 写 non-goals,并给每条配可观测的漂移信号 | PRD 的 non-goals + tripwire(如"出现仪表盘字眼就报警") → [产品判断](2026-06-20-product-judgment.md) |
| **reps 会被 AI 抄走** | 双向门交给 AI 省 reps,单向门坚持自己裁决 | 决策分诊;周期自检"最近在单向门上偷懒点头了吗" → [决策纪律](2026-06-20-deciding-with-ai.md) |
| **AI 快 ⟹ 错误注入也便宜** | 速度提上来,就把刹车(门控)同步加硬、加自动 | strict 必需检查、串行合并队列、供应链硬化 → [交付外壳](2026-06-20-delivery-harness.md) |

## 3. 元方法:怎么自己把任意"心智"造成桥

地图会过时,但造桥的方法不会。任意一条新心智,三步落地:

1. **写成"如果 X 就 Y"**:把陈述句("验证很重要")改写成带触发条件的指令("**如果**一种错误能自动检测,**就**把它设成必需门控")。没有触发条件 X 的,不是方法论,是口号。
2. **找它的"可观测触发器"**:这条规则**什么时候**该被应用?把触发器变成你能看见的信号(一个标签、一个 CI 事件、一句"我又在手动判同一件事了")。触发器越可观测,越不靠自律。
3. **尽量编译进结构,别靠记**:能写进 ruleset / lint / 模板 / 自动化的,就别留给"我会记得"。**最好的方法论最终会消失成工程实践**——因为它被结构强制了,你不用再想。

> 检验一条方法论好不好,就看它在这三步上走了多远:有触发条件吗?触发器可观测吗?能不能编译进结构、让你不用靠自律?

---

## 4. 三层之间的流向(别只单向)

- **心智 → 实践**:本篇主线(把 why 落成 how)。
- **实践 → 心智**:反过来也成立——你今天动手配 thread-resolution(实践),撞到它拦下一个自相矛盾的约定,这次经历**反哺**了"AI 评审该负责广度、人负责裁决"这条心智。**实践是心智的 reps**(见 [taste §3](2026-06-20-taste-and-judgment.md))。
- 所以别把三层当成一次性的瀑布:**它是个环——心智指导实践,实践又用真实反馈打磨心智。** 方法论是让这个环转得稳的那层。

---

## 5. 一句话总结

> **心智不落到方法论,就只是收藏的金句;方法论不落到工程实践,就只是没执行的好意。** 把每条心智写成"如果 X 就 Y",给它一个可观测触发器,再尽量编译进结构——**当一条方法论最终变成一行 ruleset、一个 lint 规则、一个模板字段时,它就从"需要你自律"升级成了"系统替你守住"。** 那一刻,心智才真正长在了你的工程里。
